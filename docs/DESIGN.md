# 多源信息核验助手 · 设计与思考文档

> 中国科学院软件研究所 · 中文信息处理实验室 Coding 测试 — 任务3
>
> 本文记录这个系统**为什么这样设计**、**关键决策的取舍**、**工程上如何实现**,以及**开发过程中真实的迭代**。代码即论据,文中关键结论都能在仓库里找到对应实现。

---

## 0. 一句话概括

输入一段说法 / 段落 / 报告,系统把它拆成**可独立核验的原子断言**,为每条断言**规划查询并路由到合适的证据源**(学术库 + 百科 + 网页),对检索到的证据做**立场研判 + 质量/时效/相关性评估**,最后用一个**可审计的打分公式**给出断言级裁决与 0–100 可信度,并生成**带编号引用的报告**。全过程通过 SSE 实时流式展示核验轨迹。

---

## 1. 问题分析:事实核验难在哪里

做这个任务前,我先想清楚"自动事实核验"区别于"让大模型判断对错"的本质难点:

1. **断言不是句子。** 一句话常包含多个事实点(如"爱因斯坦 1921 年因相对论获诺贝尔奖"其实是「1921 年获奖」+「因相对论获奖」两个断言,后者**是错的**——他因光电效应获奖)。不拆开就无法精确定位错误。
2. **LLM 的"知道"不可信、不可追溯。** 直接问模型"这句对吗"会得到一个没有出处、可能幻觉的答案,与核验的初衷(可靠、可解释、有依据)背道而驰。**证据必须来自模型之外、可被点击核对的来源。**
3. **证据本身有好坏。** 同行评审论文、预印本、百科、随机网页的可信度天差地别;一条 2015 年的结论可能已被推翻。**质量与时效必须显式建模**,而不是一视同仁。
4. **结论要有"不确定"这一档。** 现实中大量断言是"证据不足"或"来源互相矛盾"。一个只会输出真/假的系统是不诚实的。

这四点直接决定了系统的形态。

### 设计哲学:把人类核查流程显式化 + 人机各司其职

核心信念是——**不要让一次 LLM 调用替代整个核查过程**,而是把一名严谨研究者做事实核查的步骤拆解成显式、可观测、可审计的流水线;在每一步里,让 LLM 只做它擅长的语言判断,把"打分"这种需要稳定、可复现的工作交给确定性代码。

```
            人类核查者怎么做                    本系统对应阶段
  ┌───────────────────────────────┐   ┌────────────────────────────┐
  │ 这段话到底在主张什么?         │ → │ ① 断言抽取(原子化+消解指代)│
  │ 该去哪查、查什么关键词?       │ → │ ② 检索规划(按类型路由)     │
  │ 翻资料                         │ → │ ③ 多源检索(并发+去重)      │
  │ 这条资料支持还是反对?可信吗? │ → │ ④ 证据研判(立场+质量+时效) │
  │ 综合判断,有多少把握?         │ → │ ⑤ 裁决聚合(可审计打分)     │
  │ 写出带引用的结论               │ → │ ⑥ 生成报告(编号引用)       │
  └───────────────────────────────┘   └────────────────────────────┘
```

## 2. 系统架构

```
                       ┌──────────────────────────────────────────┐
   文本 / URL / 文件 ──▶│  inputs.py  统一归一化为待核验纯文本        │
                       └───────────────────┬──────────────────────┘
                                           ▼
   ┌───────────────────────────  pipeline.py 编排(异步)  ───────────────────────────┐
   │                                                                                   │
   │  ① claims.py     LLM 结构化输出 → List[Claim]  (type / checkworthiness / span)    │
   │        │  按断言并发 ↓                                                            │
   │  ② planner.py    LLM 生成差异化查询 + 按 ClaimType 路由到证据源                    │
   │  ③ retrieve.py   asyncio 并发跑 (源 × 查询),按 url 去重                           │
   │  ④ assess.py     LLM 批量判立场/相关性  ⊕  确定性代码算 质量/时效 → weight         │
   │  ⑤ aggregate.py  支持/反驳权重 → verdict + credibility(0–100)+ confidence        │
   │  ⑥ report.py     编号引用 + 整体加权可信度 + LLM 总结 → Markdown                  │
   │                                                                                   │
   └───────────────────────────────────┬───────────────────────────────────────────┘
                                        ▼  每步 emit 事件
                       ┌──────────────────────────────────────────┐
                       │  main.py  FastAPI:POST /verify 起任务      │
                       │           GET  /stream/{id}  SSE 推送轨迹   │
                       └───────────────────┬──────────────────────┘
                                           ▼
                       React 前端:实时轨迹 + 可信度仪表 + 断言卡片 + 证据冲突三栏 + 引用

   证据源(可插拔 adapter,统一 search(query)->[Evidence] 接口,均无需 API key):
     Wikipedia · arXiv · Semantic Scholar · Crossref · DuckDuckGo Web
```

## 3. 关键设计决策(取舍)

### 3.1 断言抽取:原子化、自包含、可核验性打分
`claims.py` 让模型把文本拆成**原子断言**,并强制三件事:
- **消解指代**:把"它/该模型/这项研究"替换成具体对象,使断言脱离上下文仍完整——否则检索查询无从生成。
- **类型标注**(统计/因果/定义/研究结论/时间/归属/一般事实/观点):类型直接决定下一步路由到哪些源。
- **可核验性评分** `checkworthiness`:过滤纯观点与修辞,把检索预算集中在"值得且可被外部证实"的断言上。

### 3.2 证据源路由:按断言类型分流
`planner.py` 不是把所有查询撒向所有源,而是:
- 研究结论 / 统计 / 因果 / 归属类 → 学术库(arXiv + Semantic Scholar + Crossref)+ 通用源;
- 定义类 → 优先 Wikipedia;
- 其余 → 通用网页 + 百科。

并由 LLM 生成 **3 条差异化英文查询**(核心概念 / 反面证据 / 具体实体),英文关键词对学术与网页检索召回更好,且**刻意包含"找反面证据"的角度**,避免确认偏误。

### 3.3 ⭐ 人机分工:LLM 判语义,代码算分数(本系统最重要的设计)
`assess.py` 把"评估证据"拆成两半:
- **LLM 负责**它擅长的 NLI 式判断:立场(支持/反驳/中立)、相关性、立场置信度、一句话理由。一次调用批量判完一条断言的全部证据。
- **确定性代码负责**不该交给模型随机性的打分:
  - **来源质量** `quality`:同行评审 0.85 > 百科 0.70 > 预印本 0.60 > 网页 0.40,再按引用数对数微调;
  - **时效** `recency`:按发表年份在 20 年窗口内线性衰减,**设 0.3 下限**,避免"老但经典"的奠基工作被时效抹杀。

这样打分**可复现、可审计、可解释**——评审者能逐项看到每条证据的相关/质量/时效/置信四个分量,而不是一个黑箱数字。

每条证据的综合权重:
```
weight = relevance × quality × (0.6 + 0.4 × recency) × stance_confidence
```
(recency 用 0.6–1.0 的阻尼系数,而非直接相乘,同样是为了不过度惩罚老文献。)

### 3.4 ⭐ 可信度评分:方向 × 强度
`aggregate.py` 的核心公式:
```
support_ratio = 支持权重 / (支持权重 + 反驳权重)
strength      = min(1, 总权重 / 参考权重)          # 证据"量"是否充分
credibility   = 50 + (support_ratio − 0.5) × 100 × strength
```
直觉:可信度同时由**两个独立维度**驱动——
- **方向**(`support_ratio`):证据整体偏支持还是反驳;
- **强度**(`strength`):有多少高质量相关证据。

一条只有一个弱网页支撑的断言,`strength` 小,可信度停在 50 附近(=不确定);多个强一致来源则把分数决定性地推向 0 或 100。这恰是人类核查者权衡的两件事,也让"证据不足"自然落在 50 附近而不是被误判为真或假。

### 3.5 五档裁决,而非真/假二分
`supported / refuted / partially_supported / conflicting / insufficient`。其中:
- **conflicting**:当支持与反驳权重都不可忽略(少数方 ≥ 35% 且总量足够)时显式标注"高质量来源互相矛盾,需人工判读"——诚实地暴露分歧,而不是强行二选一。
- **insufficient**:总权重低于阈值即判证据不足,可信度记 50。

## 4. 工程实现要点

| 关注点 | 做法 |
|---|---|
| **结构化 LLM 输出** | 不靠"请输出 JSON"祈祷,而是用 tool_choice 强制工具调用,入参即 JSON Schema,模型不合规会自动重试(`llm.py`)。 |
| **异步并发** | 断言之间、(源 × 查询)之间全部 `asyncio` 并发,信号量限流;整体耗时≈最慢的单条链路而非求和。 |
| **流式可观测** | 每个阶段 `emit` 结构化事件,经 SSE 实时推给前端——既是"Agent 执行轨迹"展示,也是可解释性的一部分。 |
| **容错** | 单个证据源 / 单条查询失败被吞掉,绝不拖垮整条断言;Semantic Scholar 共享池 429 自动退避后降级返回空。 |
| **可插拔证据源** | 所有源实现统一 `Source` 协议,注册表加一行即接入新源——扩展性的实证。 |
| **多输入格式** | 文本 / URL(抓正文)/ 文件(.txt/.md),在 `main.py` 按 content-type 分流。 |
| **反馈闭环** | `/api/feedback` 记录用户对每条裁决的 👍/👎,为后续来源权重 / prompt 调优留出接口。 |

## 5. 迭代过程(真实开发记录)

这一节如实记录开发中**踩到的坑和对应修正**——迭代能力体现在这里。

1. **`temperature` 报 400。** 首次调用模型即失败,抓取响应体发现 `temperature is deprecated for this model`(opus-4-8)。→ 从所有请求载荷移除该参数。*教训:不要假设 API 形状,先用最小请求探活。*
2. **JSON body 与文件上传冲突。** `/api/verify` 同时声明 Pydantic body 和 `Form/File` 参数,导致 JSON 请求被 FastAPI 当成表单,一律 400。→ 改为读原始 `Request`,按 `content-type` 手动分流 JSON / multipart。
3. **DuckDuckGo 返回 202 挑战页。** POST `html.duckduckgo.com` 拿到的是无结果的落地页。→ 实测发现 **GET 才返回真正的无 JS 结果页**,改用 GET 并解码 `uddg=` 重定向链接。
4. **Semantic Scholar 偶发 429。** 无 key 共享池限流。→ 加指数退避重试,失败则降级返回空,保证不阻塞流水线。
5. **时效过度惩罚经典论文。** 初版 `weight` 直接乘 recency,导致 2017 年的《Attention Is All You Need》被压分。→ 改用 `0.6 + 0.4×recency` 阻尼 + 0.3 下限。
6. **真实样例验证有效性。** 用"爱因斯坦因相对论获 1921 诺奖"测试,系统正确把"因相对论"判为**反驳(0 分)**(实际为光电效应),整体可信度被拉到 ~27/100——证明断言级拆分确实能定位到子句级别的错误。

## 6. 局限与未来工作

- **检索深度**:目前用摘要/片段判立场,未抓全文;对需要细读方法或数据的断言可能欠深。可加全文抓取 + 段落级证据定位。
- **中文证据源偏少**:学术源以英文为主,跨语言靠 LLM 对照;可接入中文百科 / 知网 / 微信公众号等扩充中文场景。
- **可信度阈值是工程经验值**:`参考权重`、各档比例阈值可由带标注的核验数据集校准。
- **反馈闭环目前仅记录**:已留接口,未来用反馈在线调整来源权重与检索策略(任务建议的"用户反馈驱动优化")。
- **证据冲突的更强可视化**:现以三栏支持/反驳/中立呈现,可进一步做时间轴 / 来源可信度散点。
